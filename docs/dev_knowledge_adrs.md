# 🏛️ ShiftLink Architecture Decision Records (ADR) & 개발 지식 저장소

본 문서는 프로젝트 아키텍처 설계 중 내려진 **핵심 의사결정(ADR)**과 그 기술적 배경(백엔드/데이터베이스 공부용 실무 지식)을 영구 보존하는 문서입니다.

---

## 📌 ADR-001: 백엔드(Spring Boot) vs 데이터베이스(PostgreSQL) 기능별 처리 속도/비용 우위 비교

> **💡 핵심 아키텍처 철학:** 
> "DB의 장점인 인덱싱 속도와 탐색 기능은 극한으로 뽑아먹되, 무거운 동기화/비즈니스 처리 로직은 모두 백엔드로 이관하여 보이지 않는 데드락(Deadlock)과 쓰기 병목(Write Penalty)을 차단한다."

대규모 트래픽 처리 환경에서 통신 비용 낭비를 줄이고 극한의 안정성을 달성하기 위해 확립된 **기능별 처리 우세(Dominance) 기준표**입니다.

### 📊 기능별 최적 계층 매핑 테이블

| 기능 분리 (Feature) | 가장 우세한 계층 | 사용 기술 / 기법 | 최상위 속도 및 비용 효율이 발생하는 사유 (Why?) |
| :--- | :---: | :--- | :--- |
| **대용량 조건 검색 및 필터링** | **DB (압승)** | B-Tree 인덱스 (`INDEX`) | 애플리케이션으로 수만 개를 끌고 와서 `for/loop`로 찾으면 메모리 초과(OOM)와 기가급 비용이 터집니다. DB가 자랑하는 인덱싱 탐색 고속도로(B-Tree)를 타면 최단거리로 `O(log N)` 속도로 정답만 빼냅니다. |
| **다중 데이터 결합 (관계 연결)** | **DB (압승)** | 조인 연산 (`JOIN`) | 백엔드(JPA)에서 `N+1` 문제로 쿼리 수십 번 쏘는 통신 비용보다, DB 내부 메모리에서 한 방에 머지 조인(Merge Join)하여 하나의 예쁜 표로 응답해 주는 것이 네트워크 I/O 비용상 100배 효율적입니다. |
| **통계 및 집계 렌더링 연산** | **DB (압승)** | 그룹 및 합계 (`GROUP BY`, `SUM`) | `BigDecimal` 수천 개를 배열로 통신 받아 Kotlin 서버에서 더하게 두지 마십시오. RDBMS 자체가 집계에 최적화된 C언어 기반 엔진입니다. **서버가 받을 최종 응답은 "집계가 완료된 숫자 단 1개(단일 행)"**여야 속도가 극대화됩니다. |
| **복잡한 비즈니스 계산식** | **Backend (압승)**| Service, `BigDecimal` | 주휴수당 발생 산식, 3.3% 과세, 4대 보험 등 "예외가 많고 자주 룰이 바뀌는 구조"는 버전 관리(Git)와 테스트 코드(TDD) 보장이 완벽한 백엔드 앱 계층에서 처리해야 '유지보수 야근 비용'이 극적으로 줄어듭니다. |
| **데이터 연쇄 동기화 (캐시 업데이트)** | **Backend (압승)**| 비동기 리스너 (`@EventListener`) | **[치명적]** DB `TRIGGER`를 쓰면 안 보이는 '락(Lock)'이 걸려 유저의 글 등록(INSERT) 속도 전체가 마비될 수 있습니다! 메인 스레드는 DB에 저장만 던지고 '0.1초' 만에 유저에게 응답한 뒤, **서버 뒤편의 이벤트 스레드가 백그라운드에서 여유롭게 통계 캐시를 동기화**하는 방식이 장애 없는 모던 시스템의 기초입니다. |

### 🚨 결론: ShiftLink 프로젝트의 실제 DB 튜닝(다이어트) 적용 내역

위 기준에 따라 주환님과 에이전트가 협업해 걷어낸 **오버엔지니어링(다이어트) 아키텍처 통폐합 내역**입니다.

1. **테이블 파편화 병합:** 1:1 매핑으로 존재 가치 없이 조인 성능 스캔만 갉아먹던 알림(`notification`) 테이블을 완전 폐기하고 `users` 본체 테이블 컬럼으로 직접 통폐합시켰습니다.
2. **정산 테이블 단일화:** 월간 테이블 + 주간 테이블을 따로 두지 않고 **`period_settlements` (통합 정산 캐시 테이블)** 단 하나로 병합하여 관리 포인트를 압축했습니다. (`period_type` 하나로 화면 스위칭 지원)
3. **트리거 완전 폐지:** 캐시 금액 깎는 트리거를 DB에서 모두 삭제하고, 앞서 표에서 증명한 'Spring Boot 백엔드 비동기 이벤트 큐' 방식으로 100% 이관시켰습니다.
4. **인덱스 다이어트:** 조회 성능 최적화가 필수적인 대시보드 렌더링에 한해서만 `work_logs (user_id, start_time DESC)` 단 하나의 무결점 복합 인덱스 지정만을 허용했습니다. 이로써 `INSERT` 속도 지연(Write Penalty)이 걸리는 부작용을 원천 봉쇄했습니다.

## 📌 ADR-002: 프론트/백엔드 완벽 분리 및 무상태(Stateless) 인프라 채택

> **💡 핵심 아키텍처 철학:** 
> "쿠버네티스(EKS)의 장점(트래픽 오토스케일링)을 100% 흡수하기 위해, 서버는 화면과 상태를 렌더링하지 않는 완벽한 '무상태 기계'로 전락시킨다."

* **의사결정 배경:** 모바일 애플리케이션(Flutter), 관리자 웹, 프론트엔드 웹(Next.js) 등 클라이언트가 다변화될 때 백엔드가 화면(JSP, SSR)을 렌더링하면 코드를 완전히 재작성해야 하는 막대한 중복 비용 및 버전 관리 문제가 발생함.
* **해결책 (REST API + Stateless):** Spring Boot 백엔드는 클라이언트의 종류에 상관없이 오직 순수 데이터(JSON)만 반환하도록 개발(`@RestController` 기반). 서버 내부에 유저의 세션이나 업로드 파일(디스크) 상태를 일절 저장하지 않고, AWS S3나 분산 캐시(Redis)로 모두 이관함.
* **이점 달성:** 향후 Nginx를 통한 Blue/Green 무중단 배포 시, 구버전 서버가 꺼지고 신버전 서버가 켜지는 찰나의 스위칭 순간에도 유저 세션 끊김이나 파일 분실이 없는 퍼펙트한 무중단 로드밸런싱 아키텍처 기반이 마련됨.
