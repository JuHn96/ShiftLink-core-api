# 🤖 AI 협업 및 프롬프트 문제 해결 일지 (AI Collaboration Log)

이 문서는 개발자 박주환이 AI 에이전트와 페어 프로그래밍하며 발생한 커뮤니케이션 문제, 아키텍처 제안, 그리고 룰 위반 시 궤도를 수정한 **실무적 주도 역량**을 기록하는 공간이다.

---

## 📅 [2026-03-26] Phase 1: AI의 '오염된 캐시 덮어쓰기' 실수 교정 및 환경변수(Security) 보안 선행 주도

### 1. 💡 Issue & Context (당시 상황)
* **상황:** Phase 1 마일스톤인 'Core API 뼈대 초기화' 후, AI가 `README.md` 대문에 완료 체크(`[x]`)를 덮어쓰는 과정에서 에러 발생.
* **문제 발생:** AI가 과거 대화에서 기억하던 구버전 캐시(Cache) 상태로 전체를 덮어써서, 개발자가 수동으로 직접 설계해 두었던 마일스톤 명세(`DB 연동 분리 등`)가 구버전으로 롤백(삭제)되어 버리는 대참사 발생.

### 2. 🗣️ User's Prompting & Solution (개발자의 개입 및 제안)
* **제안 1 (오류 지적 및 롤백 지시):** "음... 규칙이 조금 바뀌었네? DB 연동 자체를 Phase 2로 미루려고 한 거야? 몽고DB까지 쓴다는 걸 잊었어?"라며 AI의 아키텍처 망각과 파일 덮어쓰기 실수를 즉각 지적함.
* **제안 2 (재발 방지 룰 수립):** 단순 원상복구 처리에 그치지 않고, "ai_context 규칙에 이런 거 좀 더 확실하게 실수 없이 할 수 있게 적을 게 없을까?"라며 선제적으로 AI에게 재발 방지 룰 제정을 요구함.
  👉 **결과:** AI 행동 강령(`1_ai_action_guidelines.md`)에 **'덮어쓰기 전 최신화 검증 (Stale Cache Prevention)'**이라는 강력한 절대 룰이 신설되는 결과를 도출해 냄.

### 3. 🛡️ Advanced Architecture & Security (아키텍처 및 보안 주도)
* **제안 3 (DDD 폴더 구조화 선행):** AI가 Git Commit을 서두르자, "그 전에 작업 폴더를 역할별로 미리 나누어서 나중에 유지보수를 위해 좀 편하게 배치하고 싶은데"라며 **도메인 중심 설계(DDD)** 디렉토리 12개 뼈대 선행 구축을 완벽히 리드함.
* **제안 4 (Git 유출 해킹 방어):** Git 초기화 직전, "여기에선 .env나 properties 필요 없나?"라는 백엔드 본질적이고 예리한 질문을 핵심 타이밍에 던짐.
  👉 **결과:** AI가 해킹 유출 위험을 인지하고, 섣부른 커밋을 즉각 멈춘 뒤 `.gitignore` 보안 룰 추가 및 `application.yml` + `.env` 비밀번호 변수 바인딩 구조를 먼저 완벽히 방어(세팅)하는 성과를 도출함.

### 4. 📝 Lesson Learned (회고)
* AI는 코드를 빠르게 생성하지만, 기존 코드의 최신화 상태를 실시간으로 인지하지 못하는 '캐시 오염' 약점이 있다. 파일 덮어쓰기를 지시할 때는 무조건 최신 원본 파일 조회를 룰(Rule)로 세워야 한다.
* 프로젝트를 처음으로 커밋(저장)하기 직전에는, 반드시 민감 환경변수 보안(`.env`)과 향후 수만 줄의 코드를 유지보수하기 위한 폴더 아키텍처(DDD 등)를 먼저 챙기는 휴먼의 설계적(Architect) 직관이 필수적이다.

## 📅 [2026-03-27] Phase 1 꼬리잡기: AI의 '.gitignore 와일드카드 오버블로킹' 버그 적발 및 템플릿 구출

### 1. 💡 Issue & Context (당시 상황)
* **상황:** AI가 민감 정보 보안을 위해 `.gitignore`에 `.env.*` 와일드카드를 과도하게 적용하여 깃허브 유출을 원천 차단했다고 보고함.
* **문제 발생:** `git commit` 직전 휴먼 개발자가 코드 리뷰 중, 깃허브에 팀원 협업 가이드로 무조건 올라가야 할 빈껍데기 파일인 **`.env.example`** 마저 해당 와일드카드 블랙리스트에 쓸려들어가 억울하게 차단된 치명적인 오용 상태임을 적발함.

### 2. 🗣️ User's Prompting & Solution (개발자의 개입 및 해결 주도)
* **오류 지적 및 심층 리뷰:** "혹시 `.env.example` 파일 깃허브에 안 올라갈 텐데? 깃이그노어를 보니까?"라며 AI가 놓친 정규식/와일드카드 패턴의 치명적인 허점을 예리하게 꿰뚫어 지적함.
* **해결 로직 주도:** AI는 즉각 자신의 패착(과잉 방어)을 인지하고, Git의 독특한 예외 부정 문법(`!`)을 역으로 활용해 `!.env.example` 코드를 밑바닥에 추가함으로써, 꼭 필요한 템플릿 파일만 깃허브 업로드를 극적으로 허용시키는 방어 로직을 도출함.

### 3. 📝 Lesson Learned (회고)
* 아무리 AI가 완벽하게 룰을 짜주었다고 호언장담하더라도, **엔터키(Commit)를 치기 전 마지막 1초 전까지 이그노어 리스트를 사람의 눈으로 교차 검증(Cross-check)** 해내는 휴먼 아키텍트만의 디버깅 직관력은 절대 100% 무인 자동화로 대체될 수 없음을 깨달았다.

## 📅 [2026-03-31] Phase 2: DB 오버엔지니어링 다이어트 및 기반 아키텍처 역할 분담 리드

### 1. 💡 Issue & Context (AI의 오버엔지니어링 폭주)
* **상황:** Phase 2 (DB 모델링) 진행 중, 대시보드 차트 렌더링 성능을 극도로 높이기 위해 AI가 **DB 트리거(DB Trigger)와 1:1 파편화 테이블, 방대한 인덱스**를 강력하게 제안함.
* **문제 발생:** AI는 단순히 '조회 속도' 결과값에만 몰입하여, 실무 운영 시 발생할 치명적인 `INSERT/UPDATE` 락(Lock) 병목이나 데드락(Deadlock) 유지보수 비용 등, 백엔드 전체 생태계의 부하(Write Penalty)는 전혀 고려하지 않는 기계적인 극단적 최적화를 전개함. 

### 2. 🗣️ User's Prompting & Solution (DB 다이어트 지시 및 아키텍처 주도)
* **제안 1 (오버엔지니어링 적발):** 개발자가 즉시 "DB에서 조인, 트리거, 인덱스를 너무 과다하게 쓰면 오히려 쓰기 속도 등 DB 쪽에서 느려지고 위험해진다. 적절한 밸런스 선에서 불필요하게 쪼개진 테이블들은 싹 다 통폐합해라." 라며 AI의 맹목적인 과유불급 설계를 강력히 제동 걺.
* **제안 2 (테이블 압축 통폐합 주도):** 개발자의 이 단 한 줄의 피드백으로 인해 구체적인 Action이 연쇄 도출됨.
  👉 쓸데없이 조인 비용만 먹던 `user_notification_settings` 알림 테이블이 통째로 파기되고 `users` 본체로 흡수(Merge)됨.
  👉 파편화되려던 주간/월간 정산표가 `period_settlements` 단일 테이블 구조로 군더더기 없이 압축 다이어트 성공.
  👉 위험한 DB 트리거들이 싹 다 파기되고, 그 역할을 Spring Boot의 안전하고 가벼운 백그라운드 이벤트 리스너(Backend Async Event)로 이관시키는 이상적인 아키텍처 역할 분담 설계 완료.

### 3. 🛡️ B2C 상용 스케일의 [소셜 로그인] 및 [무결성 결함] 핀포인트 적발
* **제안 3 (통합 인증 아키텍처 리드):** "난 자체 아이디 로그인이랑 소셜(카카오/네이버) 로그인을 섞어 쓸 건데?"라며 AI가 자체 로그인 전용으로만 짜둔 B2B급 조잡한 설계를 정면 돌파함.
  👉 **결과:** AI가 즉시 소셜 유저 에러 방지용 `Null` 유연성 룰 추가, `(auth_provider + provider_id)` 복합 유니크 제약 구조를 만들어내는 등 완벽한 B2C 하이브리드 인증 설계를 토해냄.
* **제안 4 (최소 필수 식별자 점검):** 수백 개의 텍스트가 오가는 난전 속에서도 "가만, 가입 필수 최소 요건에 생년월일이 빠진 거 아냐?"라며 AI가 가장 놓치기 쉬운 도메인 본질적 식별자(`birth_date`) 결함을 마지막의 마지막에 정확히 핀포인트로 잡아내어 관통 교정함.

### 4. 📝 Lesson Learned (회고 및 ADR 확립)
* AI에게 '성능 최적화'라는 추상적 지시를 통제 없이 내리면, 수단과 방법을 가리지 않고 무리한 하드웨어 부하(트리거 연쇄) 로직을 짜버린다. AI에게는 항상 **"과도하지 않은 적정선의 밸런스 브레이크와 유지보수성"**이라는 인간 고유의 운영 휴리스틱(Heuristics)을 계속 주입해 주어야 앱이 터지지 않는다.
* 이 치열한 오버엔지니어링 검증 과정을 영구적으로 박제하기 위해, 업계 표준 문서인 **ADR(아키텍처 의사결정 기록, `docs/dev_knowledge_adrs.md`)** 포맷을 이 프로젝트에 전격 신설 도입하여, "DB vs Backend의 기능별 처리 속도/비용 우위 비교" 지식 모델을 지성 자산화(Asset) 하는 엄청난 성과를 거두었다.
